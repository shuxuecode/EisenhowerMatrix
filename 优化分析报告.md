# HTML 优化分析报告

## 文件概况

| 文件 | 行数 | 定位 |
|------|------|------|
| `index.html` | 1741 行 | 功能完整的开发版（JSONL 分文件存储） |
| `1.html` | 1612 行 | 精简的候选版（JSON 单文件存储） |

---

## 一、代码架构

| 问题 | 文件:行号 | 说明 |
|------|-----------|------|
| 两个文件冗余并存 | — | ~90% CSS 和 JS 逻辑重复，应合并为一个版本 |
| HTML 象限四重复制 | L848~927 | Q1~Q4 的 DOM 结构完全一样，应 JS 动态生成 |
| CSS 颜色硬编码 | 全局 | `rgba(255,255,255,0.1)`、`#ffd93d` 等出现几十次，应用 CSS 变量 |
| 按钮样式重复 | CSS | `.settings-btn` 和 `.trash-btn` 几乎相同却写了两份 |
| 蒙层样式重复 | CSS | `.trash-overlay` 和 `.settings-overlay` 也是两份 |
| 全局变量污染 | JS | `tasksCache`、`fileShas`、`apiToken` 等全部挂在 window，缺少模块封装 |

---

## 二、渲染性能

| 问题 | 文件:行号 | 说明 |
|------|-----------|------|
| 全量 `innerHTML` 渲染 | 两者 render() | 每次操作整个四象限 + 统计栏 + 回收站全部重新构建 DOM |
| 重复遍历数组 | 两者 render() | 每个象限 `tasks.filter()` 一次，总共 4 次，外加统计又遍历 |
| 频繁 `getElementById` | 全局 | 同一元素每次函数调用都重新查询，不缓存引用 |
| 事件反复绑定 | 两者 `bindQuadrantDropEvents()` | 用 `el.ondragover = ...` 覆盖式绑定，每次 render 后都重新执行 |
| `Date.now()` 做 ID | 两者 addInlineTask | 极快连续添加时会 ID 碰撞（虽然概率低） |

---

## 三、数据安全

| 问题 | 文件:行号 | 说明 |
|------|-----------|------|
| GitHub Token 明文存 localStorage | 两者 `saveConfig()` | XSS 攻击可窃取，应至少用 sessionStorage 或提示用户风险 |
| Token 验证无最小权限 | 两者 apiHeaders | Token 拥有仓库全部权限，没有 scope 检查，建议文档提示用户创建最小权限 Token |
| 组件已有 XSS 防护 | 两者 `escapeHtml/escapeAttr` | 这部分做得正确，但字符串拼接仍有风险点 |

---

## 四、数据一致性 & 错误处理

| 问题 | 文件:行号 | 说明 |
|------|-----------|------|
| 静默吞错误 | `index.html:1300-1303` | `clearDone()` 中 `catch (e) {}`——保存失败用户完全不知道 |
| 先更新 UI 再存数据 | 两者 | 所有操作都是先 `render()` 再 `await save`，save 失败后 UI 和存储不一致 |
| **index.html** 无冲突重试 | `index.html:1107-1129` | JSONL 分文件写没有 sha 冲突重试，多设备同时操作会丢数据 |
| **1.html** 有冲突重试更好 | `1.html:1026-1056` | `saveFileWithRetry` 实现了 sha 冲突后重新获取再写入 |

---

## 五、经验性可优化点

| 问题 | 说明 |
|------|------|
| 大量 console.log | `index.html` 中每个 API 调用、每个操作都有调试日志，生产环境应去除（`1.html` 已清理） |
| `var` 而非 `let/const` | 全部使用 `var`，应统一为 `const`/`let` |
| 单文件混合 | HTML/CSS/JS 全在 1700+ 行的一个文件中，可读性和维护性差，建议拆分为独立文件 |
| 无 service worker / PWA | 纯在线应用，离线完全不可用，可加 Cache API 兜底 |
| 回收站无自动清理 | 删除任务永久存在直到手动清空，无 TTL 机制 |
| `loadingPulse` 是假动画 | 进度条永远是 CSS 动画循环，不反映真实加载进度 |
| 移动端适配不充分 | 只有一个 `max-width:768px` 的断点，4 个 400px 高的象限在小屏体验很差 |

---

## 六、优化建议（按优先级排序）

1. **保留一个版本**：`1.html` 的数据方案（JSON + 冲突重试）优于 `index.html`（JSONL 分文件无重试），以它为基底合并
2. **引入 CSS 变量**：解决颜色重复和主题维护问题
3. **用 JS 动态生成四个象限**：从一份配置数据循环创建 DOM，减少 200+ 行重复 HTML
4. **改为增量更新而非全量 innerHTML**：toggleTask 只改一个元素的 class，不要重绘所有
5. **Token 加风险提示**：在设置面板标注「Token 会存储在浏览器本地，请使用最小权限的 fine-grained token」
6. **把 save 逻辑统一加上冲突重试**：index.html 的 `saveFileContent` 需要加上 `1.html` 的 `saveFileWithRetry` 模式
7. **清理 console.log**：index.html 有 ~20 处调试日志
